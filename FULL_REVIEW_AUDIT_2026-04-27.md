## 1. Обзор архитектуры и бизнес-модели

ManicBot представляет собой мультитенантную SaaS-платформу для автоматизации записи в салоны красоты и для частных мастеров. Архитектура разделена на два основных компонента:
1. **Worker (Бэкенд):** Работает на Cloudflare Workers, использует D1 (SQL) для хранения данных и KV для кэширования/состояний. Обрабатывает вебхуки Telegram, Instagram, WhatsApp, а также платежи Stripe.
2. **Admin Mini-App (Фронтенд):** Построен на Next.js (Edge Runtime) и развернут на Cloudflare Pages. Использует tRPC для типизированного API и NextAuth для аутентификации.

**Бизнес-модель:** B2B SaaS с тремя тарифами (Start, Pro, Max) и 14-дневным триалом. Монетизация через Stripe (ежемесячная/годовая подписка). Присутствует система плагинов (Marketplace) для дополнительных продаж (upsell).

## 2. Отчет по безопасности (Security Report)

В ходе анализа выявлен ряд критических и средних уязвимостей, а также сильные стороны текущей реализации.

### Сильные стороны:
- **Шифрование токенов:** Токены ботов и каналов (Meta) шифруются с использованием AES-GCM + HKDF-SHA256.
- **Защита от брутфорса:** Реализована блокировка после 5 неудачных попыток входа (на 15 минут) с отправкой email-уведомления при входе с нового IP.
- **Изоляция тенантов:** В D1-запросах строго проверяется `tenant_id`, что предотвращает утечку данных между салонами.
- **Безопасное сравнение:** Использование `timingSafeEqual` для проверки секретов (вебхуки, админ-ключи).

### Уязвимости и векторы атак:

| Уязвимость | Уровень | Описание | Варианты решения |
| :--- | :---: | :--- | :--- |
| **Утечка секретов в локальных файлах** | Высокий | Файлы `.dev.vars` и `admin-app/.env` содержат реальные 64-символьные hex-ключи (`ADMIN_KEY`, `AUTH_SECRET`). Если они совпадают с production, система скомпрометирована. | **Вариант 1:** Немедленно ротировать `ADMIN_KEY` и `AUTH_SECRET` в Cloudflare Dashboard. <br> **Вариант 2:** Добавить pre-commit хук (например, husky + gitleaks), блокирующий коммит файлов с секретами. |
| **Публичный доступ к методам поддержки** | Высокий | В `admin-app/src/server/api/routers/support.ts` ранее использовался `publicProcedure` для мутаций. (Частично исправлено, но требует проверки всех роутеров). | **Вариант 1:** Использовать строго `protectedProcedure` или `systemAdminProcedure` для всех God-mode функций. |
| **Уязвимость Rate Limiter (Fixed Window)** | Средний | Текущий D1 Rate Limiter использует фиксированное окно (fixed window), что позволяет злоумышленнику удвоить лимит запросов на границе сброса окна. | **Вариант 1:** Перейти на алгоритм Sliding Window Log в D1. <br> **Вариант 2:** Использовать встроенный Cloudflare Rate Limiting на уровне WAF. |
| **Недостаточная очистка HTML (XSS)** | Средний | Функция `sanitizeHtml` использует регулярные выражения, что допускает обход (например, через `style` атрибуты в маркетинговых шаблонах). | **Вариант 1:** Внедрить `isomorphic-dompurify` на сервере. <br> **Вариант 2:** Использовать строгий парсер (например, `sanitize-html` с жестким allowlist). |
| **Отсутствие проверки tenantId в Stripe Webhook** | Средний | В `checkout.session.completed` метаданные `tenantId` используются без проверки существования тенанта в D1. | **Вариант 1:** Добавить `SELECT id FROM tenants WHERE id = ?` перед обновлением статуса биллинга. |

## 3. Бэкенд (Cloudflare Workers & D1)

### Архитектурные решения:
- **Переход на D1:** Успешная миграция с KV на реляционную базу D1 для мультитенантности. Схема данных хорошо нормализована (таблицы `tenants`, `appointments`, `users`, `masters`).
- **Cron и Queues:** Использование Cloudflare Queues (`manicbot-tenant-cron`) для fan-out рассылки напоминаний решает проблему лимита CPU (30 секунд) при масштабировании свыше 100 тенантов.
- **Интеграции:** Поддержка Telegram, WhatsApp, Instagram и Google Calendar. Логика работы с Meta API вынесена в отдельные адаптеры.

### Зоны для улучшения:
1. **Обработка ошибок в Worker:** В `worker.js` используется глобальный `try/catch`, который возвращает `500 Internal Server Error`. Необходимо добавить более гранулярную обработку (например, `400 Bad Request` для невалидных вебхуков).
2. **Оптимизация D1:** В таблице `appointments` отсутствуют составные индексы для сложных выборок (например, по `tenant_id` + `master_id` + `date`). Это может замедлить работу расписания при росте базы.

## 4. Фронтенд и UX/UI (Next.js Edge)

### Текущее состояние:
- **Ролевая модель:** Отлично реализована через `RoleContext` и `WebShell`. Интерфейс динамически адаптируется под `tenant_owner`, `master`, `support` и `system_admin`.
- **God Mode:** Наличие Command Palette (Cmd+K) и Activity Feed для системных администраторов — отличный UX-паттерн для управления платформой.
- **Onboarding:** Реализован флоу `quickOnboard`, который безопасно делегирует регистрацию бота в Worker через `POST /admin/provision`.

### Проблемы UX и маршрутизации:
1. **Костыль с маршрутизацией (Cloudflare Pages):** В `layout.tsx` используется рендеринг через query-параметр `?tab=` для обхода бага `@cloudflare/next-on-pages` с вложенными маршрутами `(dashboard)/*`. Это ломает навигацию браузера (кнопки "Назад/Вперед") и ухудшает SEO/UX.
2. **Email Verification Gate:** Жесткая блокировка дашборда до подтверждения email (`EmailVerificationGate`) создает высокое трение (friction) при онбординге.

### Варианты улучшений UX:
| Проблема | Вариант 1 (Критический) | Вариант 2 (Альтернативный) |
| :--- | :--- | :--- |
| **Маршрутизация `?tab=`** | Отказаться от Edge Runtime для админки и перейти на Node.js (Vercel/Docker), чтобы использовать полноценный App Router Next.js. | Переписать структуру папок Next.js, убрав Route Groups `(dashboard)`, сделав пути явными (`/dashboard/leads`). |
| **Жесткий Email Gate** | Разрешить доступ к дашборду (Soft Gate), но заблокировать критические действия (публикация бота, рассылки) до подтверждения email. | Добавить возможность входа через Google OAuth (уже частично реализовано), что автоматически подтверждает email. |

## 5. Маркетинг и Бизнес-модель

### Текущее состояние:
- **Модуль CRM:** В `marketing.ts` заложен мощный фундамент для CRM (сегменты, шаблоны, кампании).
- **Монетизация:** Интеграция со Stripe работает корректно, поддерживаются месячные и годовые циклы (со скидкой 20%).
- **Marketplace:** Инновационная модель плагинов (1st-party extensions) позволяет продавать дополнительные функции (upsell) прямо внутри платформы.

### Критические бизнес-риски:
1. **Маркетинговые рассылки — это заглушка (Stub):** В коде `campaignSendNow` прямо указано: *"Send-now is a stub in phase 1. Campaigns are stored but no emails/SMS are dispatched yet."* Если платформа продается с заявленной функцией рассылок, это вызовет отток клиентов (Churn), так как функционал фактически не работает.
2. **Зависимость от Meta API:** Интеграция WhatsApp и Instagram требует сложной верификации бизнеса со стороны клиента. Без автоматизированного онбординга (Embedded Signup) конверсия в использование этих каналов будет крайне низкой.
