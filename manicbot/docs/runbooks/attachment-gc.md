# Attachment GC — операционный ранбук

> Сборка мусора в R2: удаление осиротевших вложений мессенджера (картинки из
> чатов, чьи сообщения удалены). Документ для оператора (Кирилл), не для
> разработчика. CLI-команды копируются построчно. Безопасно по умолчанию —
> прод НИЧЕГО не удаляет, пока ты сам не включишь флаг.

## Что это и зачем

Когда сообщение в мессенджере удаляют, его картинка в R2 (`manicbot-assets`)
остаётся. Cron-фаза `attachmentGc` (раз в 24 ч на тенанта — крон тикает каждые
15 мин `*/15 * * * *`, но окно фазы `PHASE_WINDOWS.attachmentGc = 24h` пропускает
лишние тики) находит такие осиротевшие объекты и — если включено удаление —
чистит их. Ключи content-addressed и привязаны к тенанту
(`t/<tenant>/chat_attachment-<hash>`).

**Сейчас режим — DRY-RUN (по умолчанию).** Флаг `ATTACHMENT_GC_DELETE`
отсутствует → фаза только ЛОГИРУЕТ, что удалила бы, и НИЧЕГО не трогает в R2.
Включение удаления — твоё ручное решение ПОСЛЕ просмотра dry-run логов.

## Шаг 1 — почитать dry-run логи (несколько дней)

Событие в логах: **`cron.attachment_gc.dryrun`**. Где смотреть:

```bash
cd manicbot/
npx wrangler tail --format pretty | grep attachment_gc
```

(или Cloudflare dashboard → Worker `manicbot` → Logs / Observability, фильтр по
`cron.attachment_gc`).

Поля события:

| Поле                  | Что значит                                                        |
| --------------------- | ----------------------------------------------------------------- |
| `deleted`             | сколько объектов БЫЛО БЫ удалено (в dry-run — кандидаты на снос)   |
| `kept`                | сколько ключей оставлено, т.к. на них ещё ссылается живое сообщение |
| `totalCandidates`     | всего уникальных ключей-кандидатов в этом тике (≤100 сообщений)    |
| `orphanCount`         | сколько из них — реальные сироты (без живых ссылок)               |
| `sampleKeys`          | первые ≤10 ключей-сирот (для глаз)                                |
| `samples`             | те же ключи с возрастом: `[{ key, ageSec }]`                       |
| `sampledBytes`        | примерный объём по выборке (≤10 HEAD-ов); `null`, если недоступно  |
| `sampledBytesPartial` | `true`, если сирот больше, чем попало в выборку                    |

**На что смотреть перед включением:**
- `deleted`/`orphanCount` стабильны изо дня в день, без подозрительных всплесков.
- `sampleKeys` — это правда мусор (старые `chat_attachment-*`), а не что-то живое.
- `kept` растёт вместе с трафиком — значит ref-count работает и бережёт ссылки.
- `ageSec` у сирот заметно больше 7 дней (`> 604800`) — grace соблюдается.

## Шаг 2 — включить удаление (go-live)

Когда dry-run логи выглядят здраво — включить флаг **одним** из способов:

**A) Через wrangler.toml (версионируется, аудируется).** В секции `[vars]`:

```toml
[vars]
# ... существующие vars ...
ATTACHMENT_GC_DELETE = "1"
```

затем задеплоить:

```bash
cd manicbot/
npx wrangler deploy
```

(или закоммитить в `main` — CI задеплоит сам за ~7 мин).

**B) Через Cloudflare dashboard (мгновенно, без деплоя).**
Workers & Pages → `manicbot` → Settings → Variables and Secrets →
добавить переменную `ATTACHMENT_GC_DELETE` = `1` → Save.

## Шаг 3 — подтвердить, что заработало

На следующем запуске фазы (когда её 24-часовое окно истечёт — крон проверяет
каждые 15 мин) в логах:
- событие сменится с `cron.attachment_gc.dryrun` на **`cron.attachment_gc`**;
- `deleted` теперь = РЕАЛЬНО удалённые объекты;
- message станет `Attachment GC: deleted N orphan object(s), kept M referenced`.

Если удаление падает по конкретному ключу — будет отдельный warn
`attachment_gc_delete_failed` с ключом и ошибкой (фаза не падает целиком).

## Откат (rollback)

Безопасно вернуться в dry-run в любой момент:

- **способ A:** убрать строку `ATTACHMENT_GC_DELETE` из `[vars]` (или
  поставить `"0"`), `npx wrangler deploy`;
- **способ B:** в dashboard удалить переменную (или поставить `0`), Save.

На следующем запуске фазы (см. 24-часовое окно выше) событие снова станет
`cron.attachment_gc.dryrun`, удаления прекратятся. Уже удалённые объекты не
восстановить — но удаляются только сироты без живых ссылок, старше 7 дней.

## Что гарантировано (безопасность)

- ✅ **Dry-run по умолчанию.** Без флага `ATTACHMENT_GC_DELETE=1` ни один
  объект не удаляется.
- ✅ **7-дневный grace.** Кандидаты — только сообщения, удалённые (soft-delete)
  больше 7 дней назад.
- ✅ **Изоляция тенантов.** И поиск сирот, и подсчёт ссылок идут
  `WHERE tenant_id = ?`. Чистка одного тенанта НИКОГДА не трогает объекты
  другого (есть тест `tenant isolation` в `test/attachment-gc.test.js`).
- ✅ **Ref-count склонен ОСТАВЛЯТЬ.** Подсчёт живых ссылок через `LIKE` может
  только пере-считать (ошибиться в сторону «оставить») — он никогда ложно не
  пометит ещё-ссылаемый ключ как сироту.
- ✅ **Лимит на тик.** ≤100 soft-deleted сообщений за один запуск, чтобы не
  жечь CPU.
