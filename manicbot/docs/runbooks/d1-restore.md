# D1 restore — операционный ранбук

> Что делать, когда D1 в дыму. Этот документ предназначен для оператора
> (Кирилл), а не для разработчика. CLI-команды можно скопировать
> построчно. Без программирования.

## Когда применять этот ранбук

- D1 база повреждена, недоступна, или вернула не те данные после
  миграции / некорректного UPDATE.
- Нужно поднять копию прод-данных на staging для дебага.
- Проверить, что бэкап вообще работает (раз в квартал — рекомендация).

## Что у нас есть в наличии

- **R2-бакет `manicbot-archive`**, префикс `backups/`. Внутри:
  - `backups/daily/<ISO-таймштамп>.ndjson.gz` — снапшоты каждые 6 ч,
    храним 30 дней
  - `backups/weekly/<год>-W<неделя>.ndjson.gz` — недельные снапшоты,
    храним 1 год
- **Лог `d1_backup_log`** в самой D1 — список всех успешных и
  упавших бэкапов
- **Скрипт `scripts/restore-d1.mjs`** — оркестратор. Использует
  `wrangler` под капотом, ничего вручную делать не надо

## 30-секундная процедура «восстанови последний бэкап в прод»

```bash
cd manicbot/
node scripts/restore-d1.mjs --latest
```

Скрипт:
1. Найдёт самый свежий `backups/daily/*.gz` в R2
2. Скачает, разожмёт
3. Покажет таймстамп бэкапа и количество таблиц / строк
4. Спросит подтверждение?  **Нет**, сразу выполняет. Если хочешь
   сначала посмотреть — добавь флаг `--dry-run`
5. Применит `INSERT OR REPLACE` против прод-D1 батчами по 500 строк
6. Скажет «DONE — N rows restored»

## Полный список вариантов

```bash
# Перечислить все доступные снапшоты, не делая ничего:
node scripts/restore-d1.mjs --list

# Восстановить КОНКРЕТНЫЙ снапшот (например, четверг ночью прошлой недели):
node scripts/restore-d1.mjs --key backups/daily/2026-05-20T00-00Z.ndjson.gz

# Сухой прогон — покажет первые 5 SQL-команд, ничего не пишет:
node scripts/restore-d1.mjs --latest --dry-run

# Восстановить в локальную dev-D1 вместо прода (для проверки бэкапа):
node scripts/restore-d1.mjs --latest --local
```

## Что делает restore (и чего не делает)

✅ Применяет `INSERT OR REPLACE` для каждой строки каждой таблицы из
бэкапа. Существующие строки с тем же PK перезаписываются.

✅ Безопасен для повторного запуска: можно перезапускать сколько угодно
раз, эффект тот же.

❌ **Не удаляет** строки, которых в бэкапе нет. Если за время после
бэкапа в прод-БД были созданы новые строки — они **останутся**.
Это сознательное решение: «полный wipe» — отдельная операция, чтобы
случайно не уничтожить новые данные.

❌ **Не накатывает схему**. Если бэкап старый и в новой схеме появились
колонки — отсутствующие колонки получат `NULL`. Если откатываешь после
неудачной миграции — сначала `wrangler d1 migrations apply manicbot-db
--remote` чтобы синхронизировать схему, потом restore.

## Авария «не могу даже подключиться к prod D1»

Если D1 базы вообще нет (удалена через dashboard, например):

1. Создать новую D1: Cloudflare dashboard → Workers & Pages → D1 → Create
   - Name: **точно** `manicbot-db` (Worker связан по имени)
2. Накатить миграции:
   ```bash
   cd manicbot/
   npx wrangler d1 migrations apply manicbot-db --remote
   ```
3. Восстановить данные:
   ```bash
   node scripts/restore-d1.mjs --latest
   ```

## Если в R2 нет ни одного бэкапа

Тогда дамп взять негде, ситуация фатальная. Превентивно убедиться, что
этого никогда не случится:

- Каждое утро (или раз в неделю) проверять дату последнего успешного
  бэкапа. Два способа:
  - быстрый список снапшотов из R2: `node scripts/restore-d1.mjs --list`;
  - точный лог в самой D1 (таблица `d1_backup_log`):
    ```bash
    npx wrangler d1 execute manicbot-db --remote \
      --command "SELECT finished_at, kind, status, row_count FROM d1_backup_log ORDER BY finished_at DESC LIMIT 5"
    ```
- Бэкап идёт каждые 6 ч. Если последний успешный (`status='success'`)
  старше ~6-7 ч — в системе проблема: смотри `wrangler tail` / Cloudflare
  Logs на `worker.d1Backup` (ошибки рана) и строки `status='failed'`
  в `d1_backup_log` с `error_message`.

## Если `wrangler` ругается на auth

Скрипт использует `npx wrangler` под капотом. wrangler читает токены
из:
- `CLOUDFLARE_API_TOKEN` env-переменной, или
- `~/.wrangler/config/default.toml` (после `wrangler login`)

Сделать `wrangler login` один раз в терминале, дальше скрипты сами
работают. Если в проде хочешь автоматизировать — выдай Cloudflare
API-токен с правами «D1: Edit», «R2: Read», «Workers Scripts: Read».

## Проверка целостности бэкапа (раз в квартал)

```bash
# Шаг 1 — список того, что есть в R2
node scripts/restore-d1.mjs --list

# Шаг 2 — сухой прогон последнего, проверить что header читается
node scripts/restore-d1.mjs --latest --dry-run

# Шаг 3 — полный restore в локальную dev-D1 (НЕ в прод!)
node scripts/restore-d1.mjs --latest --local

# Шаг 4 — поднять wrangler dev, открыть admin-app, проверить что
#         данные на месте: тенанты, мастера, услуги, последние записи
npm run dev
```

Если все 4 шага зелёные — бэкап жив. Запись в календарь на следующий
квартал.
