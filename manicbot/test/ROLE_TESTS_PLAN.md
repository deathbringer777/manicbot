# План тестов ролей и сценариев

## Цель

Убедиться, что назначение мастера возможно **только внутри одного тенанта**: пользователя можно добавить мастером в бота только если он уже заходил в **этого** бота (есть запись в `u:{chatId}` этого тенанта).

## Чеклист сценариев

### 1. Назначение мастера (tenant scope)

- [ ] **Сценарий 1.1** Админ тенанта A в боте тенанта A добавляет пользователя по @username — пользователь ранее нажал /start в боте A → успех.
- [ ] **Сценарий 1.2** Админ тенанта A добавляет пользователя по числовому ID; этот ID есть в u: тенанта A (пользователь заходил в бота A) → успех.
- [x] **Сценарий 1.3** Админ тенанта A вводит числовой ID пользователя, который **никогда** не заходил в бота A (нет в u: тенанта A) → ошибка «Сначала пользователь должен зайти в этого бота» (реализовано: проверка getUser(ctx, masterId) в STEP.ADD_MASTER).
- [ ] **Сценарий 1.4** Админ тенанта A добавляет по пересланному сообщению пользователя, который не в u: тенанта A → ошибка.
- [ ] **Сценарий 1.5** Список мастеров и запись master:* изолированы по тенанту (разные префиксы) → уже покрыто в masters-kv.test.js.

### 2. Роли и доступ

- [ ] **Сценарий 2.1** В тенантном боте только админ/tenant_owner может добавлять мастеров (isAdmin).
- [ ] **Сценарий 2.2** getRole в тенантном боте возвращает master только если есть запись в master:{cid} этого тенанта.
- [ ] **Сценарий 2.3** Пользователь — мастер в тенанте A, заходит в бота тенанта B (никогда не был мастером в B) → в B он клиент.

### 3. Ошибки и граничные случаи

- [ ] **Сценарий 3.1** Добавление мастера при достижении лимита мастеров по плану → ошибка с текстом лимита.
- [ ] **Сценарий 3.2** Добавление уже существующего мастера → «Этот мастер уже добавлен».
- [ ] **Сценарий 3.3** Ввод невалидного ID (ноль, отрицательный, не число) → ошибка.
- [ ] **Сценарий 3.4** @username пользователя, который ни разу не писал этому боту → подсказка «пусть откроет бота и /start».

### 4. Конфликты

- [ ] **Сценарий 4.1** Один и тот же Telegram-аккаунт мастер в тенанте A и тенанте B — допустимо (разные салоны).
- [ ] **Сценарий 4.2** /grant_master в главном боте с указанием tenantId (system_admin) — отдельный flow, не меняем; в тенантном боте /grant_master недоступен (только админка «Мастера»).

## Реализация проверки в коде

- [x] В потоке ADD_MASTER перед `saveMaster(ctx, masterId, ...)` проверять в тенантном боте: `getUser(ctx, masterId)` не null. Иначе отправлять сообщение `adm_master_must_use_bot_first` и не вызывать saveMaster. (message.js, проверка только при ctx.tenantId.)
- [x] Юнит-тесты: test/role-master-tenant.test.js — изоляция мастеров по тенанту, getUser по тенанту, один chatId может быть мастером в двух тенантах.
